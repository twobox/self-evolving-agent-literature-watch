# Agent Instructions

本仓库用于持续执行 Self-Evolving LLM Agent 相关论文的发现、筛选、去重和状态跟踪。

## 任务入口

执行任何论文监控任务前，必须依次阅读：

1. `GUIDE.md`
2. `STATE.md`
3. `candidates.md`
4. `SCREENING_LOG.md`
5. `queries.md`

如任务是完整运行一次监控报告，还必须使用根目录中的 `RUN_REPORT_TEMPLATE.md`。这是唯一规范的运行报告模板。

## 完整运行要求

当用户要求“执行一次完整监控”“运行论文监控”或同等含义的任务时，必须完成 `GUIDE.md` 中定义的“单次完整运行事务”，不得只在聊天中返回结果，也不得只新增一份运行报告。

完整运行至少应当：

- 新建 `runs/YYYY/YYYY-MM-DD-*.md`；
- 合并更新 `candidates.md`；
- 按需更新 `SCREENING_LOG.md`；
- 更新 `STATE.md`；
- 更新 `runs/README.md`；
- 核对报告统计与实际记录一致。

即使本轮没有发现有效候选，也要生成真实的运行报告并更新运行状态。

## 数据与状态约束

- 同一论文在 `candidates.md` 中只能存在一条规范记录。
- 优先使用 arXiv ID、DOI 或 OpenReview ID 作为稳定 `Paper ID`。
- 同一论文的 arXiv、OpenReview、会议和期刊页面必须合并。
- 不得把“本次首次发现的旧论文”描述为“近期新发布论文”。
- 不得回写历史运行报告；论文后续变化应记录在新的运行报告及候选总表中。
- “本次出现类型”和“仓库当前状态”必须使用不同字段。
- 只把容易重复出现、已实质判断排除的文章写入 `SCREENING_LOG.md`，不要保存全部检索噪声。
- `STATE.md` 和 `runs/README.md` 使用固定结构；更新时应保留既有标题、字段名和表格列，不得另建并行状态区或自定义索引格式。
- 不得重新创建 `ignored.md`、`templates/run-report.md` 或其他与规范文件并行的别名入口。

## 工作边界

除非用户明确要求，本仓库中的监控运行不负责：

- 下载或保存论文 PDF；
- 检查代码、模型和数据集；
- 生成完整论文笔记；
- 修改外部论文笔记仓库；
- 为达到数量要求而降低筛选标准。

如发现适合生成正式笔记的论文，只更新其候选状态和后续动作。
