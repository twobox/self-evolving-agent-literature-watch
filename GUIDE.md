# Self-Evolving LLM Agent 相关论文定期监控指南

## 0. 运行方式

本任务用于持续发现 Self-Evolving LLM Agent 及其相邻方向的新论文和此前遗漏的重要论文。

基线建立是一个可跨多次运行推进的阶段。首次运行应启动基线、建立初始候选池，并在 `STATE.md` 中记录各方向的实际覆盖状态；后续运行继续补全尚未充分覆盖的方向，直到主要方向达到足够覆盖。

完成首次运行后，常规运行仍应同时包含一次增量监控和一个有限范围的历史补漏任务。不得把本次首次发现的旧论文描述为近期新发布论文。

每次运行开始时记录：

- 本次运行日期；
- 上次运行日期；
- 增量监控时间范围；
- 本次历史补漏方向或检索范围；
- 是否已访问 `candidates.md`、`SCREENING_LOG.md` 和历史运行记录。

## 1. 监控目标

持续监控 Self-Evolving LLM Agent 及其相邻研究方向，重点关注 Agent 如何通过以下机制积累经验并提升后续能力：

- 记忆与经验积累；
- 上下文、Prompt 或 Harness 更新；
- 轨迹总结、修正与复用；
- 工具使用策略优化；
- 强化学习与反馈学习；
- 合成任务或训练环境生成；
- Agent 代码、策略或模型参数更新；
- 递归自我改进。

同时关注能够支撑该方向的相关研究，包括模型适应能力与能力边界、Agent 能力诊断、Agent 学习与自我改进评测，以及与持续学习、测试时学习和跨任务泛化相关的基础方法。

优先发现提出新方法或新系统、给出新实验结果或研究结论、能够补充现有分类、能够加入横向实验对比，或能够填补已有候选池研究空白的工作。

仅存在关键词重合，但研究问题与 Agent 的经验积累、持续改进或后续能力提升没有实质联系的论文，不纳入有效候选。

## 2. 时间范围

默认检索当前日期向前滚动三年内：

- 首次公开的论文；
- 在此期间发生重要更新的论文；
- 此前遗漏但仍具有较高研究价值的论文。

历史论文不要求一次找齐，可以通过多次运行按研究方向、关键词、会议、时间段和来源逐步补充。超出三年的论文原则上不进入常规候选表；若较早论文被近期工作反复引用，并且是理解当前方向所必需的基础工作，可以作为例外收录，并标记为“基础工作补充”。

## 3. 搜索来源与检索方法

每次运行结合以下四种路径：

1. **主题检索**：使用 Self-Evolving Agent、Self-Improving Agent、Agent Learning from Experience 等直接主题词；
2. **机制检索**：围绕 Agent Memory Update、Experience Reuse、Harness Optimization、Trajectory Refinement、Test-Time Learning、Agent Reinforcement Learning、Tool-Use Learning 和 Synthetic Task Generation 等具体机制；
3. **论文关系扩展**：从核心论文的参考文献、被引论文、相似论文及相关作者后续工作中扩展；
4. **来源扫描**：按需扫描 arXiv、OpenReview、主要会议和期刊的近期论文列表。

不得只依据标题中是否出现 Self-Evolving、Memory、Reflection 或 Agent 等词汇，应判断论文是否真正涉及 Agent 的经验积累、持续改进或后续能力提升。

每次历史补漏应轮换一个研究子方向、关键词组、时间段、会议或来源，避免重复执行完全相同的查询。实际使用的查询和实际检查的来源必须记录在本次运行报告中。

监控阶段默认不下载论文 PDF，也不收集代码、模型和数据集。

## 4. 去重与筛选

### 4.1 去重

运行前必须读取 `candidates.md` 和 `SCREENING_LOG.md`，按规范化论文标题判断论文是否已经出现。规范化时忽略大小写、首尾空格、连续空格和常见标点差异；规范化后的标题相同，视为同一论文。

### 4.2 实质相关性判断

对候选论文至少回答三个问题：

1. **什么发生了更新？** 例如记忆、上下文、Prompt、Harness、轨迹、工具策略、模型参数、Agent 代码或训练任务；
2. **更新是否被保存并用于后续行为？** 例如影响后续回合、后续任务、同类任务或跨任务表现；
3. **论文是否提供能力提升证据？** 例如成功率提高、跨任务泛化增强、工具使用改善、长期性能提升或成本下降。

仅进行同一问题内多次重试、一次性反思、普通多采样、只在当前上下文中重新规划，或没有持久化与复用机制的临时修正，通常不视为直接的 Self-Evolving Agent 工作。

强化学习、合成数据、工具使用和模型训练论文，只有在其明确服务于 Agent 的持续改进、经验积累或后续能力提升时，才作为直接候选。

### 4.3 推荐判断

筛选结论分为：

- **推荐收录**：与主题直接相关，方法、实验或结论具有明显价值；
- **暂时观察**：具有一定相关性，但证据、完整度、发表状态或补充价值仍需进一步确认；
- **忽略**：仅有关键词重合，缺乏实质联系，或当前证据明显不足。

推荐结论综合考虑相关程度、创新性、实验质量、代表性、证据可信度及对现有候选池的补充价值。

### 4.4 重要更新判断

已有论文只有出现以下变化时，才作为重要更新输出：

- 从预印本变为会议或期刊正式接收；
- 方法发生实质变化；
- 增加重要实验、数据或分析；
- 主要结论发生变化；
- 标题、作者或论文定位发生明显变化。

普通版本号变化、排版或文字修改、参考文献更新、引用次数变化，以及不影响方法和结论的小幅修订，通常不视为重要更新。

## 5. 候选论文分类

对推荐收录和暂时观察的候选，记录：

- **论文类型**：方法、系统、评测、诊断、综述或基础理论；
- **演化对象**：记忆、上下文、Prompt、Harness、轨迹、工具策略、模型参数、Agent 代码或训练任务与环境；
- **能力提升范围**：单次任务内、跨回合、同任务族、跨任务、跨领域或尚不明确；
- **证据可信度**：高、中或低。

分类用于后续整理和比较，不要求为了填满字段而进行过度检索。

## 6. 输出格式

每次运行以紧凑的候选论文表格为主要输出，便于在桌面端和手机端快速横向筛选，不生成完整论文笔记。报告开头简要说明运行日期、增量范围、历史补漏方向、实际使用的查询、实际检查的来源和是否完成仓库比对。

推荐收录和暂时观察使用统一的六列表格：

| 论文 | 发表情况 | 相关方向 | 核心贡献 | 判断理由 | 当前状态 |
| --- | --- | --- | --- | --- | --- |

其中：

- **论文**：完整标题并链接官方页面；
- **发表情况**：首次公开日期及当前能够确认的发表状态；
- **相关方向**：合并表达论文类型、演化对象和能力提升范围；
- **核心贡献**：用一至两句话概括问题、方法和主要贡献；
- **判断理由**：说明研究价值、证据和与现有候选池的关系；
- **当前状态**：记录“推荐收录”或“暂时观察”，并用括号注明本次出现类型。

已有论文重要更新单独列出论文、原状态、当前状态和变化内容。被忽略的论文只记录标题和忽略原因，避免未来重复筛选。

引用次数为可选信息，仅在能够可靠获取时写入判断理由，并注明来源和查询日期。

## 7. 单次运行规模与结束条件

单次运行默认最多初步检查约 50 篇论文，优先展示不超过 20 篇有效候选，但不得为了达到数量要求降低筛选标准。

当预设关键词组、研究方向和主要来源均已完成检查，达到候选数量上限，或连续多组查询未发现新的有效候选时，可以结束本次运行。符合要求的论文较少时应按实际结果结束。

“推荐收录不超过 10 篇”仅作为主表展示建议，不影响实际推荐判断；超出部分可进入补充表或后续批次。

## 8. 运行结束摘要

每次运行结束时说明增量时间范围、历史补漏方向、已完成的主要关键词组和来源，并汇总：

- 实际检查论文数；
- 去重后论文数；
- 推荐收录；
- 暂时观察；
- 忽略；
- 新近发布；
- 历史补充；
- 基础工作补充；
- 重要更新。

所有统计只基于本次实际保存或核对过的记录。没有发现符合要求的论文时，应如实说明。

## 9. 单次完整运行事务

一次完整监控运行必须作为一个一致的仓库事务执行：

1. 读取 `GUIDE.md`，确认当前规则版本；
2. 读取 `STATE.md`，确定上次运行日期、增量起点和下一轮建议；
3. 读取 `candidates.md` 和 `SCREENING_LOG.md`，建立去重基线；
4. 读取 `queries.md`，确定本轮增量查询和一个有限的历史补漏范围；
5. 执行检索、核验、去重和实质相关性判断；
6. 根据根目录 `RUN_REPORT_TEMPLATE.md` 新建 `runs/YYYY/YYYY-MM-DD-NN.md`；
7. 将推荐收录和暂时观察的有效候选合并更新到 `candidates.md`；
8. 将容易重复出现且已完成实质判断的排除项按需更新到 `SCREENING_LOG.md`；
9. 按 `STATE.md` 的固定字段和表格结构更新上次运行、增量截止日期、覆盖进度、待跟踪论文和下一轮建议；
10. 按 `runs/README.md` 的固定列更新运行索引；
11. 检查候选总表没有重复论文，状态字段互不混淆，报告统计与实际写入记录一致。

`SCREENING_LOG.md` 和 `RUN_REPORT_TEMPLATE.md` 分别是历史筛选记录与运行报告模板的唯一规范入口。不得创建或维护 `ignored.md`、`templates/run-report.md` 或其他并行别名文件。

`STATE.md` 与 `runs/README.md` 的标题、字段名和表格列属于执行契约。运行时只更新字段值和新增规范表格行，不得另建表达相同含义的状态区或索引格式。

不得只生成聊天回复或只新增运行报告。即使没有发现有效候选，也应生成真实报告并更新 `STATE.md` 与运行索引。

历史运行报告原则上不可改写。论文发生后续变化时，应在新报告中记录，并更新当前状态文件，而不是修改旧报告中的历史判断。

## 10. 状态字段约定

### 10.1 本次出现类型

描述论文为何出现在某次运行中：

- 新近发布；
- 历史补充；
- 基础工作补充；
- 已有论文重要更新。

### 10.2 仓库当前状态

描述论文的当前筛选结论：

- 推荐收录；
- 暂时观察。
