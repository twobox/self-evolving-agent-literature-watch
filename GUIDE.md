# Self-Evolving LLM Agent 论文候选收集指南

## 0. 仓库定位

本仓库用于发现可能相关且值得后续阅读的论文，并维护候选论文的基本信息。

监控阶段只做轻量判断：根据标题和摘要判断论文是否可能与目标主题相关，并补充与本次动作相匹配的客观信息。仓库不负责论文精读、方法评价、实验判断或质量分级。

## 1. 三种独立运行

仓库只包含三种运行方式：

1. **近期论文监控**；
2. **历史优质论文搜索**；
3. **已有候选信息更新**。

一次运行只执行一种动作。不得把近期监控、历史搜索和已有信息更新捆绑为一次“完整运行”，也不得在执行某一动作时顺带完成另一动作。

## 2. 监控主题

持续关注 Self-Evolving LLM Agent 及相邻研究方向，包括但不限于：

- Agent 经验积累与复用；
- 记忆更新与长期记忆；
- Agent Skills、Harness、Prompt 或上下文优化；
- 轨迹、工作流与工具使用学习；
- Agent 强化学习与持续学习；
- 自我改进、递归改进及相关评测。

主题范围用于检索和标题、摘要相关性判断，不要求在监控阶段证明论文满足严格的 Self-Evolving Agent 定义。

## 3. 共同判断规则

### 3.1 相关性判断

只阅读论文标题和摘要，判断其是否可能与监控主题相关。相关即可进入候选池，不进一步判断：

- 具体更新了什么机制；
- 更新是否持久化；
- 是否真正提升后续能力；
- 方法是否创新；
- 实验是否充分；
- 证据是否可信或具有代表性。

### 3.2 不建立评价状态

仓库不使用以下状态或类似变体：

- 推荐收录；
- 暂时观察；
- 待复核；
- 忽略；
- 质量较强或较弱；
- 分数、等级或优先级。

普通搜索噪声直接跳过，不建立长期排除记录。

### 3.3 去重

运行前读取 `candidates.md`。同一论文只保留一条记录，通常按规范化标题去重；规范化时忽略大小写、首尾空格、连续空格和常见标点差异。

发现已有论文时，本次动作直接跳过，不顺带更新其信息。已有信息更新只能由专门的“已有候选信息更新”动作执行。

## 4. 近期论文监控

### 4.1 目标

发现最近一段时间新公开、与监控主题可能相关的论文。

### 4.2 时间范围

优先使用用户指定的时间范围。用户未指定时，从 `STATE.md` 中最近一次近期监控覆盖日期之后开始，截止当前日期；尚无记录时选择一个有限的近期范围并在报告中明确说明。

### 4.3 主要来源

- arXiv；
- OpenReview；
- 主要会议近期接收列表；
- 必要时使用作者、实验室或研究机构页面核实机构信息。

### 4.4 收集信息

对新增论文记录：

- 论文标题与链接；
- 根据标题和摘要生成的一句话介绍；
- 首次公开日期和当前可确认的发表状态；
- 作者机构情况，使用简短机构名称，例如“清华大学；腾讯”。

作者机构只作为提供给用户的背景信息，不打分、不分级，也不作为硬性排除条件。

### 4.5 边界

近期论文监控不查询引用情况，不搜索历史论文，不批量核查已有候选的发表状态。

## 5. 历史优质论文搜索

### 5.1 目标

围绕一个指定研究方向、时间段、会议、期刊或来源，补充此前遗漏的较有影响力的历史论文。

### 5.2 主要来源

- 高水平会议和期刊论文列表；
- 引用数据库；
- 相关综述和代表性论文的参考文献；
- 核心论文的被引论文和相似论文；
- 代表性作者的相关工作。

### 5.3 收集与筛选

先根据标题和摘要判断主题相关性，再使用正式发表情况和引用情况帮助寻找较有影响力的论文。

这些信息只作为客观资料记录，不建立统一阈值、质量分数或等级。历史搜索不要求调查作者机构。

历史论文的发表和引用情况可简洁写入 `首次公开或发表状态` 单元格，例如：

```text
NeurIPS 2024；引用 320（Semantic Scholar，2026-06-27）
```

引用信息必须注明来源和查询日期；无法可靠获取时可以只记录发表情况。

### 5.4 边界

历史优质论文搜索不扫描近期论文，不更新已有候选信息，也不建立待复核列表。

## 6. 已有候选信息更新

### 6.1 目标

只核查 `candidates.md` 中已经存在的论文，并直接更新可确认的基本信息。

### 6.2 可更新内容

- 首次公开时间；
- 是否仍为预印本；
- 已确认的会议或期刊；
- 正式论文页面；
- 标题变化；
- 作者机构信息的明确变化。

本动作默认不刷新引用情况。若未来需要批量刷新引用，应作为单独任务明确执行。

### 6.3 输出

直接更新候选表对应单元格。运行报告只列出实际发生变化的论文以及更新前后的信息；不列出无变化论文，不添加“已更新”“无变化”或“无法确认”等状态列。

### 6.4 边界

不得发现或加入新论文，不重新判断已有论文的主题相关性，不进行论文质量评价。

## 7. 候选池格式

`candidates.md` 使用以下固定表格：

| 序号 | 论文标题 | 一句话介绍 | 首次公开或发表状态 | 作者机构情况 | 用户备注 |
| ---: | --- | --- | --- | --- | --- |

字段要求：

- **序号**：按表格顺序连续编号；
- **论文标题**：完整标题，并链接到可靠论文页面；
- **一句话介绍**：只根据标题和摘要客观概括；
- **首次公开或发表状态**：记录首次公开日期、当前正式发表状态；历史搜索时可同时记录引用情况；
- **作者机构情况**：近期监控时填写简短机构名称；其他动作没有必要时保持原值或留空；
- **用户备注**：仅由用户维护，Agent 不得填写、修改、清空或移动。

更新任何已有行时，必须原样保留 `用户备注`。

## 8. 运行报告

每次运行在 `runs/YYYY/` 新建：

```text
runs/YYYY/YYYY-MM-DD-NN.md
```

Run ID 等于不含 `.md` 的文件名。报告保持简短，不设置固定的推荐、观察、忽略或待复核章节。

### 8.1 近期论文监控报告

只记录：

- 运行日期与监控范围；
- 实际查询和来源；
- 本次新增论文；
- 新增数量。

### 8.2 历史优质论文搜索报告

只记录：

- 本次研究方向与实际覆盖范围；
- 实际查询和来源；
- 本次新增论文；
- 新增数量。

### 8.3 已有候选信息更新报告

只记录：

- 本次检查范围；
- 实际发生的信息变化；
- 更新数量。

没有新增或没有变化时，也应生成简短报告并如实记录零结果。

## 9. 单次运行事务

一次运行按以下步骤执行：

1. 确认用户指定的运行类型；
2. 读取 `GUIDE.md`、`candidates.md`、`STATE.md`、`queries.md` 和 `RUN_REPORT_TEMPLATE.md`；
3. 只执行该运行类型对应的搜索或核查；
4. 按需更新 `candidates.md`，并保留所有用户备注；
5. 新建一份简短运行报告；
6. 只更新 `STATE.md` 中与本次动作对应的记录；
7. 在 `runs/README.md` 追加运行索引；
8. 核对 Run ID、报告路径、新增或更新数量一致。

## 10. 工作边界

除非用户明确要求，本仓库不负责：

- 下载或保存论文 PDF；
- 生成完整论文笔记；
- 检查代码、模型或数据集；
- 深入阅读论文方法和实验；
- 建立质量评分或推荐系统；
- 修改外部论文笔记仓库。
