# 可复制的运行提示词

仓库中的 `AGENTS.md`、`GUIDE.md` 和 `STATE.md` 已经保存固定规则。日常提示词只需要说明：**本轮有哪些变量、工作边界是什么、完成要求是什么**，不必重复整套监控指南。

## 1. 推荐：自动执行一次完整运行

适合绝大多数日常使用。首次运行时，Agent 应根据 `STATE.md` 启动基线阶段并建立初始候选池；后续运行执行增量监控，同时继续补全尚未覆盖的基线方向。

```text
请在当前仓库执行一次完整的论文监控运行。

先读取 AGENTS.md，并严格按照 GUIDE.md、STATE.md 和 queries.md，根据当前状态确定增量时间范围和历史补漏方向。

完成后执行 GUIDE.md 规定的完整运行事务，更新所有应修改的文件。不要只在对话中输出搜索结果或只新增运行报告。

本次只维护论文监控仓库，不生成完整论文笔记，不下载论文 PDF，也不检查代码、模型和数据集。

结束时简要汇报：本次运行范围、候选统计、新增或更新的文件，以及下一轮建议。
```

这是默认推荐提示词。它把规则和状态判断交给仓库，只保留必要的工作边界。

## 2. 指定历史补漏方向

当你希望本轮重点检查某个子方向时使用。只需要替换方括号中的内容。

```text
请在当前仓库执行一次完整的论文监控运行。

先读取 AGENTS.md，并严格按照 GUIDE.md 和 STATE.md 执行：

- 增量监控范围：以上次完整运行的截止日期为起点，截至当前日期；
- 本次历史补漏方向：[填写研究子方向、关键词组、会议、时间段或来源]；
- 其他检索范围和结束条件：根据 queries.md 与仓库当前状态确定。

完成后执行 GUIDE.md 规定的完整运行事务，更新所有应修改的文件，不要只在对话中输出结果。

本次只维护论文监控仓库，不生成完整论文笔记，不下载论文 PDF，也不检查代码、模型和数据集。

结束时简要汇报本次发现、候选统计、文件修改和后续待跟踪事项。
```

示例补漏方向：

- Agent Reinforcement Learning 与跨任务经验固化
- Agent Memory 的在线更新与长期稳定性
- Harness、Prompt 与 Agent Skills 自我优化
- 合成任务、课程与训练环境生成
- Agent 自我改进评测、能力边界与安全退化

## 3. 启动基线阶段

通常默认提示词已经可以根据 `STATE.md` 识别尚未开始的基线阶段。需要明确指定首次运行时，可以使用：

```text
请在当前仓库执行首次完整监控运行，并启动基线阶段。

先读取 AGENTS.md、GUIDE.md、STATE.md、candidates.md、SCREENING_LOG.md 和 queries.md。按照 GUIDE.md 选择一个有限的基线范围，开始检查当前日期向前滚动三年的相关论文，建立初始候选池，并在 STATE.md 中记录本次实际覆盖与后续待补全方向。优先覆盖 Agent Memory、Experience Reuse、Harness 和 Agent Skills。

完成后创建首份运行报告，并同步更新 candidates.md、SCREENING_LOG.md、STATE.md 和 runs/README.md。不要只在对话中返回报告。

本次不生成完整论文笔记，不下载论文 PDF，也不检查代码、模型和数据集。

结束时汇报候选统计、主要发现、当前基线覆盖进度和下一轮建议。
```

## 4. 仅检查已有候选的重要更新

适合集中核查发表状态、方法变化或新增重要实验，不执行大范围历史补漏。

```text
请检查当前仓库候选论文是否发生重要更新。

先读取 AGENTS.md、GUIDE.md、STATE.md 和 candidates.md，按照 GUIDE.md 的“重要更新判断”核查已有候选。只记录正式接收、方法实质变化、重要实验或结论变化等重要更新，不把普通版本号或排版修改算作重要更新。

将核查结果写入一份新的运行报告，并同步更新 candidates.md、STATE.md 和 runs/README.md。没有重要更新时也要如实记录。

本次不执行大范围历史补漏，不生成完整论文笔记。
```

## 5. 只讨论或试运行，不写入仓库

需要先讨论检索策略、筛选标准或查看可能结果时，必须明确说明不修改仓库，避免被理解为完整运行。

```text
请根据当前仓库规则讨论一次论文监控方案，但不要修改任何仓库文件。

先读取 AGENTS.md、GUIDE.md、STATE.md 和 queries.md，然后说明建议的增量范围、历史补漏方向、查询组合和预计结束条件。此任务只做方案讨论，不执行完整运行。
```

## 6. 提示词设计原则

每次提示词建议只包含以下三项：

1. **本轮变量**：历史补漏方向、特殊时间范围或特定来源；
2. **工作边界**：是否生成论文笔记、下载 PDF 或检查代码；
3. **完成要求**：完整运行必须落盘并更新全部状态文件。

通常不需要在提示词里重复具体表格字段、去重规则、筛选标准和文件修改清单。这些内容应以仓库中的 `GUIDE.md` 为唯一规则来源。
